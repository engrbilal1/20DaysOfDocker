# Contributing to 20 Days of Docker

Thank you for your interest in contributing to this project! 🎉

We welcome contributions from everyone - whether you're fixing typos, adding examples, improving exercises, or suggesting improvements.

---

## 🤝 How to Contribute

### Reporting Issues

Found a bug or problem? Help us improve!

1. **Check existing issues** - Make sure it hasn't been reported
2. **Create new issue** - Include:
   - Clear title
   - Detailed description
   - Steps to reproduce (if applicable)
   - Expected vs actual behavior
   - Your environment (OS, Docker version, etc.)

### Improving Content

Want to improve existing content?

1. **Fork the repository**
   ```bash
   Click "Fork" on GitHub
   ```

2. **Clone your fork**
   ```bash
   git clone https://github.com/YOUR-USERNAME/20DaysOfDocker.git
   cd 20DaysOfDocker
   ```

3. **Create a branch**
   ```bash
   git checkout -b feature/improve-day-content
   # or
   git checkout -b fix/typo-in-readme
   ```

4. **Make your changes**
   - Edit files
   - Add examples
   - Improve explanations
   - Fix mistakes

5. **Test your changes**
   - Read through your edits
   - Verify code examples work
   - Check formatting

6. **Commit your changes**
   ```bash
   git commit -m "Improve Day X: [brief description]"
   # Good examples:
   # git commit -m "Add real-world example to Day 3"
   # git commit -m "Fix typo in Day 5 Networking section"
   # git commit -m "Clarify ENTRYPOINT vs CMD in Day 3"
   ```

7. **Push to your fork**
   ```bash
   git push origin feature/improve-day-content
   ```

8. **Create a Pull Request**
   - Go to GitHub
   - Click "New Pull Request"
   - Provide title and description
   - Explain what you changed and why

## 📋 Contribution Guidelines

### Code of Conduct

- Be respectful and inclusive
- Welcome diverse perspectives
- Focus on constructive feedback
- Help others learn

### Content Standards

When contributing content, ensure:

✅ **Accuracy**
- Information is correct
- Code examples work
- Commands are tested

✅ **Clarity**
- Language is simple and clear
- Explanations are concise
- Examples are practical

✅ **Completeness**
- Include necessary context
- Provide solutions when applicable
- Link to resources

✅ **Formatting**
- Use proper Markdown
- Include code blocks with language
- Add proper headings
- Use lists for clarity

### Writing Style

- Use second person ("you")
- Keep sentences short
- Explain concepts simply
- Include practical examples
- Use active voice

### Code Examples

Include examples when relevant:

```dockerfile
# Always include language identifier
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install -r requirements.txt

COPY . .

CMD ["python", "app.py"]
```

**Good examples:**
- Copy-paste ready
- Actually work
- Have expected output
- Include explanations

## 🔄 Review Process

When you submit a pull request:

1. **Automated checks** run
2. **Maintainers review** your changes
3. **Feedback provided** if needed
4. **Revisions made** as necessary
5. **Merged** when approved

Don't be discouraged by feedback - it helps make everything better!

---

## 💡 Tips for Success

✅ **Small changes are easier to review**
- Submit focused pull requests
- One improvement per PR
- Clear commit messages

✅ **Reference issues in PRs**
- Link to related issues
- Explain the connection
- Close issues when applicable

✅ **Be responsive**
- Reply to feedback promptly
- Make requested changes
- Ask questions if unclear

✅ **Check existing work**
- Review similar contributions
- Avoid duplicating efforts
- Build on previous work
